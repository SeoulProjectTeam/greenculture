# 서울시 빅데이터 공모전 팀 공유 문서

## 1. 프로젝트 한 줄 소개
- 사용자가 원하는 날짜의 문화행사를 추천받고, 해당 장소까지 가는 여러 이동 경로를 시간/편의성/탄소 기준으로 비교해 최적 선택을 돕는 서비스.

## 2. 해결하려는 문제
- 기존 서비스는 행사 추천과 길찾기가 분리되어 있어 실제 이동 의사결정이 어렵다.
- 단순 최단시간 경로는 탄소 절감, 따릉이 활용, 환승 편의성 같은 요소를 반영하지 못한다.
- 사용자는 "어떤 경로가 나에게 더 좋은지"를 직관적으로 이해하기 어렵다.

## 3. 우리 서비스의 핵심 가치
- 문화행사 탐색부터 이동 선택까지 하나의 흐름으로 연결.
- 빠른 길만이 아니라 친환경성과 실사용 편의성을 함께 비교.
- 데이터 기반 근거와 설명 문장을 함께 제공해 의사결정 부담 감소.

## 4. 주요 기능 (MVP 기준)

### 4-1. 문화행사 추천
- 입력: 날짜, 관심사(전시/공연 등)
- 출력: 해당 조건에 맞는 문화행사 목록
- 목적: 사용자가 갈만한 목적지를 먼저 빠르게 좁히기

### 4-2. 경로 생성 및 3가지 추천
- 빠른 경로: 이동 시간 최소
- 친환경 경로: 탄소 배출 최소
- 균형 경로: 시간과 탄소를 함께 고려한 타협안

### 4-3. 경로 비교 정보 제공
- 이동 시간
- 탄소 배출량
- 환승 횟수
- 따릉이 포함 여부
- 목적: "왜 이 경로가 좋은지"를 수치로 비교 가능하게 제공

### 4-4. 따릉이 대여소 추천
- 출발지/도착지 주변 대여소 제시
- 따릉이 연계 경로 선택 가능성 강화

### 4-5. 따릉이 반납 가능성 예측 (AI)
- 특정 시간대, 특정 대여소 기준 반납 가능성 예측
- 결과: 높음 / 보통 / 낮음
- 목적: 따릉이 경로 선택 시 실패 리스크 감소

### 4-6. 탄소 절감량 시각화
- 자가용 기준 대비 절감된 탄소량 제공
- 친환경 행동의 효과를 사용자에게 직관적으로 전달

### 4-7. 생성형 AI 기반 경로 설명
- 예: "이 경로는 이동 시간이 가장 짧습니다."
- 예: "이 경로는 탄소 배출을 크게 줄일 수 있습니다."
- 목적: 숫자 중심 화면을 이해하기 쉬운 문장으로 보완

## 5. 사용할 데이터
- 서울시 따릉이 대여/반납 이력 데이터
- 따릉이 대여소 위치 데이터(위도/경도)
- 문화행사 데이터
- 교통 경로 API 데이터(TMAP 또는 카카오)
- 교통수단별 탄소 배출 계수 데이터

## 6. 차별점 (기존 서비스 대비)
- 추천 + 이동 + 친환경 지표를 한 서비스 안에서 통합 제공.
- "최단시간 1개"가 아니라 목적이 다른 3개 경로를 동시에 제시.
- 따릉이 반납 가능성 예측으로 현실적인 이동 성공률까지 고려.
- 생성형 AI 설명으로 비전문 사용자도 결과를 이해하기 쉬움.

## 7. 기대 효과
- 시민의 문화 접근성 향상(행사 선택 및 이동 계획 단순화)
- 친환경 이동 유도(탄소 절감 행동 촉진)
- 서울시 공공데이터 활용 우수 사례로 확장 가능

## 8. 발표/심사 포인트(팀 공통 인지)
- 공공성: 시민 체감 문제 해결 + 서울시 정책 방향(탄소중립/대중교통) 정합성
- 기술성: 데이터 기반 추천, 예측 AI, 생성형 설명 결합
- 실효성: 시간/탄소 절감의 정량 결과 제시
- 확장성: 축제/관광/생활이동 등 다른 도시 서비스로 확장 가능

## 9. 팀 역할 제안
- 기획/발표: 문제정의, 사용자 시나리오, 정책 연결 메시지
- 데이터/AI: 예측 모델 개발, 성능 검증, 지표 리포트
- 백엔드: 추천/비교/계산 API, 데이터 연동
- 프론트앱: 경로 비교 UI, 설명/시각화 UX

## 10. 다음 액션 (단기)
- 핵심 시나리오 1개 확정: "날짜 입력 -> 행사 선택 -> 3경로 비교 -> 최종 선택"
- MVP 화면 와이어프레임 합의
- 실험 지표 확정: 시간, 탄소, 예측 정확도
- 중간 점검 일정 고정(예: 3일 단위)

## 11. 현재까지 진행사항(구현 현황)

### 11-1. 리포지토리 구조
- `backend/`: Spring Boot API 서버
- `frontend/`: Expo 기반 React Native 앱(안드로이드 에뮬레이터 기준)
- `docs/`: 기획/공유 문서
- `docker-compose.yml`: 로컬 PostgreSQL 실행용

### 11-2. 백엔드(완료/동작 확인)
- 문화행사 추천 API
  - `GET /api/events/recommend?date=YYYY-MM-DD&interest=카테고리`
- 경로 비교/설명 API(현재는 샘플 데이터 기반)
  - `POST /api/routes/compare`
  - `POST /api/routes/{routeAlternativeId}/explain`
- 따릉이 주변 대여소/반납 예측 API(현재는 샘플/목업 기반)
  - `GET /api/bike-stations/nearby`
  - `GET /api/bike-stations/{stationId}/return-availability`
- 탄소 계산 API
  - `POST /api/carbon/calculate`

### 11-3. 문화행사 데이터 적재(OpenAPI 연동 완료)
- 서울 열린데이터광장 `culturalEventInfo`(XML) 연동
- 관리자 임포트 API 추가(수동 적재)
  - `POST /api/admin/events/import?start=1&end=200`
  - `POST /api/admin/events/import-range?pageSize=200&maxItems=1000`
- 중복 방지: `cultcode`(HMPG_ADDR에서 추출) 기준으로 upsert
- 환경변수로 키 주입
  - `SEOUL_OPENAPI_KEY` (필수)

### 11-4. 자동 갱신(스케줄러 추가)
- 매일 지정된 시간에 OpenAPI 임포트를 자동 실행하도록 스케줄러 추가
- 기본값은 안전하게 OFF
  - `SEOUL_IMPORT_ENABLED=true`로 켜기
  - `SEOUL_IMPORT_CRON`, `SEOUL_IMPORT_PAGE_SIZE`, `SEOUL_IMPORT_MAX_ITEMS`로 조정

### 11-5. DB(PostgreSQL) 로컬 구성(완료)
- Docker Compose로 로컬 Postgres 실행
- Spring JPA(`ddl-auto=update`)로 테이블 자동 생성

### 11-6. 프론트(완료/동작 확인)
- Expo + TypeScript 앱 스캐폴드
- 하단 탭 4개 화면
  - 행사 / 경로 / 따릉이 / 탄소
- 행사 화면 UI 개선
  - 날짜: 달력(DatePicker) 선택
  - 카테고리: 드롭다운(Picker) 선택
- 안드로이드 에뮬레이터에서 백엔드 접근 기본값
  - `http://10.0.2.2:8080`

### 11-7. 로컬 실행 순서(요약)
- Postgres 실행: `docker compose -p greenculture up -d postgres`
- 백엔드 실행: `backend`에서 `.\gradlew.bat bootRun`
- (선택) 행사 임포트: PowerShell에서 `Invoke-WebRequest -Method POST ".../api/admin/events/import?..."`
- 프론트 실행: `frontend`에서 `npm run android`
